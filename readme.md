## I. Backend/Frontend асуултууд

1. CSS-ийн block, inline, inline-block утгуудын ялгаа юу вэ?
block - Parent елементийн өргөнийг шууд дүүргэнэ. Доторхи елементүүд дээр дээрээсээ давхарлагдсан байдалтай харагдана. Нэг үгээр шинэ мөрнөөс доторхи элэментүүд (children) эхэлнэ.


inline - Default, юу ч бичээгүй тохиолдолд inline гэж үзэгддэг. үгний дагуу (inline), доторхи елементүүд block шиг хүчээр мөр дүүргэхгүй, ар араасаа цуварсан маягтай байрлана. Өөрт нь width, height ямар ч нөлөөгүй (float эсвэл flex дотор байвал хамаатай), доторхи элэментүүдээс хамаарна. Дээшээ, доошоо padding/margin үйлчлэхгүй.


inline-block - энгийнээр inline + block. Хүүхдүүд нь шинэ мөрнөөс эхлэхгүй inline байж шууд parent элэментийн өргөнийг дүүргэхгүй. мөн Дээшээ, доошоо padding/margin үйлчлэхгүй гэсэн асуудал байхгүй.

3. Promise гэж юу вэ? async/await ашиглахын давуу тал юу вэ? Мөн error handling-ийн талаар бичнэ үү.


Promise - JS нь single thread буюу ганц урсгалаар явдаг хэл. Тиймээс сүлжээгээр юм татах, код ажиллаж дуусах хүртэл өөр зүйлс хийх зэрэг олон thread ажиллаж байгаа юм шиг asynchronous ажиллах боломжийг олгодог.
ExamplePromise(FunctionToCall((resolve,reject)), successCallback, failureCallback) гэсэн syntax-р бичигдэнэ. resolve() дуудагдвал successCallback ажиллана. Зүгээр дуудахгүй код execution дуусаад орхиж болно, гэхдээ тооцоллоос буцах утгыг дамжуулахад маш хэрэгтэй. reject() дуудагдвал failureCallback ажиллана. гарах алдааг .catch() дотор зохицуулна.


3 утгатай байна
-pending: Тооцоолол дуусаагүй.
-fulfilled: resolve дуудагдсаан л гэсэн үг
-rejected: reject дуудагдсан.


async/await ашиглахын давуу тал


await-г зөвхөн promise дээр ашиглана. Promise-г ямар нэг хариу өгтөл хүлээхэд ашигланаа. async кодуудын sync юм шиг (дээрээс доош дараалалтай) ажилуулах зорилготой. 


яагаад ашигладаг юэ?

- кодны гоо сайхан, уншихад амар байдалд маш хэрэгцээтэй

Syntax : 
await yamarNegPromise;
await yamarNegPromiseButsaahAsyncFunction();

async нь function нь asynchronous юмаа гэдгийг тодорхойлно нөгөөгөөр бол promise буцаадаг function юмаа гэсэн үг. Мөн дотор нь await дуудах боломж үүснэ. Үгүй бол дээрээс доош гэсэн дүрмээр код ажиллаж promise хүлээх нь боломжгүй болоод буцаад л callback hell рүүгээ орноо.

Syntax : 
async yamarNegPromiseButsaahAsyncFunction(){return Promise().then((r=>{return r;}))};


4. React-д useMemo болон useCallback hook-үүдийг ямар тохиолдолд ашигладаг вэ? Ашиглахгүй бол ямар асуудал үүсэх вэ?


2 уулаа л хэрэггүй олон re-render хийхээс сэргийлсэн зүйлс. React нь state өөрчлөгдөх болгон ахин render хийгддэг тул render хийх болгонд функц байн2 дуудагдахгүйн тулд ашигланаа.

useMemo
Syntax: let cachedValue = useMemo(() => {Ямар нэг тооцоолол, return "value"}, [хамааралтай state хувьсагчууд]); 
хамааралтай state хувьсагчууд өөрчлөгдөх тохиолдолд л ахин дуудагдаж cachedValue утга өөрчлөгдөнө. Мөн хамгийн анхны render хийгдэхэд нэг удаа дуудагдана. Тооцоолол хийх зорилгоор ашиглана. 
useEffect юм бишүү?
- Ай бас үгүйээ, useMemo дотор state утга өөрчилж болохгүй, Мөн useEffect бүтэн функцийг тэр чигт нь хадгалдаг бол useMemo зөвхөн тооцолол буцаасан утгыг л хадгалнаа. - useMemo render хийхээс өмнө дуудагдана. Мөн маш том удаан тооцоолол хийчвэл render удаад хүлээгддэг. Харин useEffect бол render хийсний дараа дуудагддаг.

useCallback
Syntax : 
const handleSubmit = useCallback((orderDetails) => {
    post('/product/' + productId + '/buy', {
      referrer,
      orderDetails,
    });
  }, [productId, referrer]);

функц хадгалах зориулалттай. Мөн ашиглаж буй эсвэл ямар нэг state-с хамааруулж ахин утгыг шинэчилдэг. дээрхи жишээ syntax тайлбар болох байх. useMemo той адил render хийх явцад дуудагдана.

5. React дээр unnecessary re-renders үүсэх нийтлэг шалтгаанууд юу вэ? Үүнийг хэрхэн засах вэ?

React нь state өөрчлөгдөх болгонд ахин render хийдэг.Энэ нь ихэнхидээ responsive, гоё feature хийхэд амар болгодог ч заримдаа бүр маш олон хэрэггүй render хийх тохиолдол руу ч оруулнаа. Товчхондоо хэрэггүй re-render үүсэж байнаа гэдэг нь ихэнхидээ бид state-ээ тааруухан зохицуулалт хийсэн байнаа гэсэн үг байдаг.

Доорхи хэдэн дүрэм-г баримтлахад ихэнхи тохиолдолуудаас сэргийлж чаднаа.

- Parent render хийхэд child бас render хийгдэнэ.
Тиймээс child component рүү дамжуулж байгаа props дээр state ашиглахгүй байх. Үнэхээр тухайн state л хэрэг болоогүй бол useMemo, useCallback props-р дамжуулах.
useMemo, useCallback ашиглавал утга өөрчлөгдөөгүй бол ахин render хийдэггүй.
- Component доторхи хувьсагч/функц болгон render хийх болгонд ахин зарлагддаг.
component доторхи хувьсагч, функц render хийх болгонд ахин зарлагддаг учир том хувьсагч, функц зэрэг зүйлсийг component-с гаргах болвол static утга болгох үгүй бол useMemo,useCallBack зэргийг ашиглах.
- useEffect гинжин хэлхээ үүсгэхгүй байх
useEffect дотор state-н утга өөрчлөх нь өөрийгөө болон өөр useEffect ахин дуудагдаж олон render хийх эсвэл бүр render loop рүү ч оруулах боломжтой.

7. Dependency Injection гэж юу вэ? FastAPI-д яаж ашигладаг вэ?

Dependency Injection нь объект эсвэл функц өөрт хэрэгцээтэй байгаа туслах хэрэгслүүдээ (dependencies) өөрөө дотроо үүсгэх/агуулах хэрэггүйгээр, харин гаднаас "тарьж" (inject) оруулж ирэхийг хэлдэг software deisgn pattern бөлгөө.

FastApi дээр Depends keyword ашиглана. Нэг зүйлийг олон ашиглах, хамаарал багасгах, бизнес логикийг бусад хамааралаас холдуулах зорилгуудаар ихэвчлэн ашиглана.

Жишээ : 
@app.get("/admin")
def admin_only(authorized: bool = Depends(RoleChecker(["admin"]))):
    return {"message": "Welcome, Admin"}

10. Python-д decorator гэж юу вэ? FastAPI-д @app.get() гэх мэт decorator ямар үүрэгтэй вэ?

decorator гэдэг нь програмчлалд function-г ажиллахаас өмнө болон дараа ажиллах логик зааж өгч буй wrapper юмаа. 
Жишээ :
def my_decorator(func):
    def wrapper():
        print("Функц ажиллахаас өмнө")
        func()
        print("Функц ажилласны дараа")
    return wrapper

@my_decorator
def say_hello():
    print("Сайн байна уу!")

say_hello()

@my_decorator гэж бичих нь say_hello = my_decorator(say_hello) гэж бичсэнтэй яг ижил утгатай.

@app.get("/") гэсэн decorator нь тухайн функцийг routing table рүү бүртгэн http get method-той холбож байгаа үйлдэл юм. Энгийнээр ийм боловч цаанаа маш олон үйлдэл хийж байгаа.


## III. Infrastructure ☁️
1. Terraform гэж юу вэ? Ашиглахын гол ач холбогдол юу вэ? Ойлгомжтой тайлбарлана уу.
Infrastructure as Code - IaC буюу дэд бүтцийг код хэлбэрээр удирдах зориулалттай хэрэгсэл/tool/ юмаа. .tf форматтай файл ашиглана. Минийхаар бол DevOps engineer-ийн Docker юмдаа. Энэ нь энгийн үгээр бол "Надад 2 CPU, 4GB RAM-тай 3 ширхэг сервер хэрэгтэй байна" гэж бичээд бусдыг нь tool-д даатган цаашаа аль нэг Cloud service provider (AWS, Azure, GCP  зэрэг) үүсгэх үйлдэлийг автомат болгож байгаа хэрэг юм.
Яагаад ашиглах хэрэгтэй юм бэ?
- Автоматжуулалт ба Хурд
Мэдээж автоматжуулж буй гол зорилго нь цаг хэмнэх болон хурд. Та 100 сервер хэр хурдан тохируулж чадах вэ?
- Multi-Cloud
зөвхөн нэг service provider-аар хязгаарлагдахгүй. Нэг код дотор AWS, Azure, Google Cloud, тэр байтугай Kubernetes-ийг ч зэрэг удирдаж болно.
- Хувилбарын хяналт (Version Control)
Нэгэнт л дэд бүтэцээ код болгочихвол яг л Git шиг version control хийх боломж үүснээ.

Орчин цагт Cloud service гарж ирсэн учираас дэд бүтэцийг автоматжуулах боломж үүссэнийг дагаж гарч ирсэн хэрэгсэл.

2. Terraform State: State файл гэж юу вэ? Remote state болон state locking ашиглахын давуу тал юу вэ?
State file нь тухайн файл болон жинхэнэ сервер 2 холбоо, одоо ямар байгаа, юунаас хараамч байгаа зэрэг мэдээлэлийг хадгалдаг. nodeJs ашигладаг хүнд бол package-lock гээд хэлчихэд бараг буруудахгүй. Гараар засах маш хэрэггүй, өөрөө авто үүсдэг.
Remote State нь өөрийн машин дээр (local) state файл ашиглахгүй гэсэн үг юм. Багаараа нэг state файлтай байх, CI/CD тохируулах зэрэгт ашиглана. Гэхдээ хамгийн чухал нь Багаараа нэг state файлтай байх.
state locking нь харин олон хүн нэг зэрэг файлын ажилуулах, засах зэргээс сэргийлсэн цоожлох механизм юм. Хэн нэгэн ашиглаж байнаа гэдгийг тодорхойлдог. 

3. IaaS / PaaS / SaaS гурвын ялгааг тайлбарлана уу.
IaaS (Infrastructure as a Service)
- Дэд бүтэцийг үйлчилгээгээр авна. Сервер компьютер түрээслэж авах бол энэ үйлчилгээ юм. OS суулгах, package manage хийх гээд бараг бүх холбоотой юмаа өөрөө тохируулж таардаг. Их эрх чөлөөг олж авах боловч маш их юм өөрөө хийх хэрэг гарна. Доод түвшний бүтээгдхүүн гэж хэлж болно.
PaaS (Platform as a Service)
Platform буюу орчин үйлчилгээгээр авах. Үйлдлийн систем, програм ажиллах орчин, сервер зэргийг бүгдийг авна. Зүгээр код тань байхад л болно гэсэн үг. Heroku, Vercel зэрэг service provider-үүд алдартай. 
SaaS (Software as a Service)
Энэ бол яг эцсийн бүтээгдхүүн үйлчилгээ болгон авч буй юм. Таньд бол өөрийн тохиргоо л хэрэгтэй. Gmail, Slack, Zoom, Jira гээд орчин цагийн ихэнхи software ийм хэлбэртэй болсон. 
4. Serverless computing гэж юу вэ?
Нэр нь Serverless болхоос энэ нь огт сервергүй гэсэн үг бас угтаа биш юм. IaaS (Infrastructure as a Service) - ийг сервер түрээслэх гэж ойлговол serverless-ийг функц түрээслэх гэж ойлгож болох юм. MicroService, API, Дата боловсруулах зэрэгт ашиглахад тохиромжтой. Олон state зохицуулах шаардлагатай, тэрэнтэйгээ уялдаж ажиллах програм бичихэд тохиромж муутай. Энгийн зураг resizer, chatbot зэрэг тодорхой нэг функцтэй зүйлс ашиглахад амар
Гол давуу талууд нь 
- ямар ч сервер зохицуулах хэрэггүй. Ийм болхоор ч serverless гэж нэрлэсэн байх.
- auto-scaling. Ачаалал, хэрэгцээгээр хэрэгтэй тооцололын хүчийг тохируулна. Ачаалал бага байсан ч, их байсан ч гацахгүйээ гэсэн үг.
- Хэрэглэснээрээ төлөх. Таны хэрэглээнээс хамаарч төлбөр бодогдоно. Маш их ачаалалтай ажиллаж байвал өндөр төлбөр гарна. Заримдаа DDos хийлгээд маш их өрөнд уначихсан хүмүүс ч байдаг. Тиймээс анхааралтай л хэрэглэх хэрэгтэй.

5. DDoS халдлагаас хэрхэн хамгаалах вэ?
DDoS (Distributed Denial of Service) нь энгийнээр дааж чадахаас нь их хүсэлтийг явуулж серверийг гацаах бүр цаашлад унтрахад хүртэл хүргэдэг нэг төрлийн халдлага юм.
Энийг anti-virus програм шиг нэг удаа тохируулаад мартдаг зүйл биш бөгөөд байнгын хяналттай байж бүрэн сэргийлэх боломжтой юм.
Хамгаалах үндсэн 2 түвшин байна
Hardware түвшинд
- Хамгаалалттай сайн сүлжээний тоног төхөөрөмж ашиглах, түүнийгээ байнга software update хийх. Хэрвээ cloud ашиглаж байгаа бол нэгээс олон дата төвтэй, түүн дээрээ хуваарилах. Мөн үүнээс хамгаалах зориулалттай cloudflare, akamai зэрэг сервисүүд ашиглах.
Hardware түвшинд асуудлыг шийдэх нь ихэнхи тохиолдолд software шийдлээс хамаагүй хурдан, ашиглалтын зардал бага байдаг.

Software түвшинд
- Хүсэлтийн агуулга, давтамж, хаяг зэргийг шалгах, түр зуур block хийх үгүй бол хар жагсаалтанд оруулах. 
- Captcha ашиглах. 
- Програмаа ачаалал даах чадвар өндөр болгох
- static файлуудад хатуу эрх тавих. Хаанаас яаж хандаж болох зэрэг. Үүнийг CDN service-р авч болно.

Хамгаалалт биш ч auto-scaling тохируулвал таны сервер ядаж амьд байж байх боломжтой.

DDos нь яг hack хийхгүй ч гэсэн доорхи зорилгуудаар ихэвчлэн ашигладаг.
- Үйлчилгээ авах боломжгүй болгох.
- Маш их төлбөрт унагах
- Жинхэнэ hack хүсэлтүүдээ нуух (Маш олон хүсэлт дундаас шүүж харах, олох боломж багасна)
- Зарим ховор тохиолдолд аль дээр hack хийчихсэн зүгээр л restart хийлгэх хэрэгтэй байх үед
